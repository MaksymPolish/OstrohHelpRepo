# OstrohHelp Web - Site Documentation & ToDo Plan

## 1. Мета сайту
OstrohHelp Web - платформа психологічної підтримки для студентів та психологів.

Основні цілі:
- безпечна автентифікація через Google OAuth;
- робота з анкетами та консультаціями;
- чат між студентом і психологом з real-time оновленням;
- стабільна робота у desktop і mobile.

## 2. Технологічний стек
- Frontend: React 19, react-router-dom, Tailwind CSS.
- API layer: Axios з JWT/refresh flow.
- Realtime: SignalR.
- Auth: @react-oauth/google.
- Build: react-scripts.

## 3. Поточна структура модулів
- App shell, routing, security context: src/App.js
- Layout (header/sidebar/footer): src/components/Layout
- Сторінки: src/pages
- API/infra сервіси: src/services
- Локалізація: src/i18n/translations.js

## 4. Що вже реалізовано
- Google OAuth login та локальна сесія користувача.
- Базова робота з профілем користувача.
- Сторінка 404.
- Список консультацій і чат-інтерфейс.
- Завантаження вкладень у чаті.
- Realtime отримання повідомлень через SignalR.
- Базова інтеграція шифрування повідомлень на фронті.

## 5. ToDo план імплементації

### P0 - Критично (завершити в першу чергу)
- Стабілізувати end-to-end flow шифрування чату з бекендом:
  - перевірити повну сумісність формату payload для POST /Message/Send;
  - підтвердити дешифрування історії (GET /Message/Recive) і realtime (ReceiveMessage);
  - уніфікувати контракт для iv/authTag у фронті та бекенді.
- Уніфікувати sender/receiver поля у всіх send-flow сценаріях.
- Додати retry/fallback UX для помилок ключа шифрування (ReceiveConsultationKey ще не прийшов).
- Показувати зрозумілі помилки користувачу без технічних деталей.

### P1 - Високий пріоритет
- Покрити ключові user flows тестами:
  - login/logout;
  - load consultations;
  - send/receive encrypted message;
  - upload attachment.
- Реалізувати стабільний статус доставки/читання:
  - mark-as-read інтеграція в UI;
  - індикатори read/unread.
- Додати pagination або lazy loading для великих історій чату.
- Покращити валідацію файлів перед upload:
  - обмеження розміру;
  - whitelist mime type;
  - зрозумілі повідомлення про помилки.

### P2 - Середній пріоритет
- Завершити модуль анкет:
  - кращий UX для заповнення;
  - фільтрація/статуси;
  - детальні сторінки з історією змін.
- Розширити модуль консультацій:
  - фільтри, сортування, стани;
  - краща навігація між активними і завершеними консультаціями.
- Додати повну сторінку ресурсів з контентом і категоріями.
- Підготувати role-based UI для HeadOfService (початок адмін-модуля).

### P2 - UI Redesign (планована зміна інтерфейсу)
- Підготувати нову дизайн-концепцію:
  - оновити візуальну систему (кольори, типографіка, відступи, іконографія);
  - узгодити єдині патерни для кнопок, інпутів, карток, модалок.
- Оновити layout додатку:
  - переробити Sidebar, Header, Footer;
  - покращити адаптивність для mobile/tablet/desktop.
- Переробити ключові екрани:
  - Login, Home, Consultations, Profile, Questionnaires;
  - уніфікувати loading/empty/error стани у всіх модулях.
- Винести повторно використовувані UI-елементи у спільні компоненти:
  - сформувати набір базових компонентів (UI kit);
  - мінімізувати дублювання стилів між сторінками.
- Провести UX-перевірку після редизайну:
  - перевірити сценарії входу, навігації, чату та заповнення анкет;
  - зафіксувати фідбек і внести правки перед релізом.

### P3 - Низький пріоритет / Підготовка до продакшену
- Налаштувати моніторинг фронтенду:
  - runtime error tracking;
  - метрики продуктивності.
- Додати CI перевірки:
  - lint;
  - build;
  - unit/integration tests.
- Провести accessibility аудит (контраст, клавіатурна навігація, aria).
- Підготувати deployment checklist і релізний процес.

## 6. Технічний борг
- Частина API-контрактів має дублікати camelCase/PascalCase.
- Потрібна централізація нормалізації DTO у сервісному шарі.
- Потрібно зменшити зв'язність UI-логіки і транспортного шару у чаті.
- Варто винести chat state у окремий hook або state container.

## 7. Критерії готовності (Definition of Done)
Задача вважається завершеною, якщо:
- реалізація покрита базовим тестом;
- немає помилок у збірці;
- UX має обробку loading/error/empty state;
- API контракт підтверджений фактичним сценарієм у dev середовищі;
- зміни задокументовані в цьому файлі.

## 8. Рекомендований порядок робіт (наступні спринти)
1. Закрити P0 encryption/send/receive стабільність.
2. Додати тести на критичні chat/auth flows.
3. Завершити mark-as-read + message status UX.
4. Виконати UI Redesign для ключових сторінок і layout.
5. Розширити анкети/консультації (P2).
6. Підготувати prod-ready контур (P3).
